    技术复盘：Chrome 扩展如何批量导出有道云笔记

    9分钟 Navyum · 技术复盘 2026年7月15日

    有道云笔记导出助手的技术目标很明确：用浏览器扩展完成认证、拉取、转换、资源本地化和 ZIP 下载，尽量不引入服务端依赖。

    整体架构

    项目基于 Chrome Extension Manifest V3。弹窗负责交互，Service Worker 负责后台导出任务，核心逻辑拆分在 src/core/ 目录下。

    模块职责
    auth读取浏览器 Cookie，确认 note.youdao.com 登录状态。
    youdao-api封装有道云笔记接口请求。
    exporter编排导出流程、进度和错误记录。
    converter将普通笔记转换为 Markdown。
    special-converter处理表格、思维导图、白板等特殊格式。
    zip-builder使用 JSZip 构建最终 ZIP。

    认证设计

    扩展不要求用户手动复制 Cookie。只要浏览器中已登录 note.youdao.com，就可以通过 Chrome cookies API 读取必要登录状态，并在请求有道接口时使用。

    这种方式降低了用户门槛，也避免用户把 Cookie 粘贴到第三方网页或命令行工具里。

    导出编排

    导出流程大致分为五步：

    1. 加载笔记本树。
    2. 按用户勾选范围收集可导出的文件。
    3. 逐个下载笔记正文和资源。
    4. 转换为 Markdown，并重写资源链接。
    5. 构建 ZIP，通过浏览器下载。

    资源本地化

    云笔记里常见的问题是图片链接依赖在线服务。导出助手会下载引用资源，保存到同级 assets 目录，再把 Markdown 中的链接替换为相对路径。这是迁移到 Obsidian 等本地工具时最关键的一步。

    特殊格式处理

    项目针对常见特殊格式提供独立转换逻辑：表格转 Markdown table，思维导图转缩进列表，Draw.io 和 Excalidraw 输出 SVG 资源并在 Markdown 中引用。

    后台任务与状态

    MV3 Service Worker 的生命周期需要谨慎处理。项目把导出状态保存到扩展本地存储，弹窗关闭后后台任务仍可继续执行，用户重新打开弹窗时可以恢复进度展示。

    为什么不做服务端

    服务端可以集中处理任务，但会引入隐私、成本和可信问题。这个项目的定位是个人数据导出工具，因此优先选择本地浏览器处理，哪怕实现上需要处理更多浏览器限制。

    后续可改进方向

    • 更细粒度的失败重试和错误分类。
    • 更完善的大文件资源限速和排队策略。
    • 更多 Markdown 兼容性测试样例。
    • 导出前后的完整性报告。
